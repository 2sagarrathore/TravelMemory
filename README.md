# Travel Memory

A full-stack MERN travel journal application. This fork extends the original [UnpredictablePrashant/TravelMemory](https://github.com/UnpredictablePrashant/TravelMemory) with a production-style AWS deployment.

---

## Deployment

The application runs across two AWS EC2 instances behind an Application Load Balancer, with MongoDB Atlas as the database layer and Cloudflare DNS for the custom domain.

### Architecture

```
User
 └──> Cloudflare DNS (saagarsaga.store)
       └──> AWS Application Load Balancer (ap-south-1)
             ├──> EC2 Instance 1 (Ubuntu 24.04, t2.micro)
             │     ├─ Backend  : Node.js + Express, port 3000 (PM2)
             │     ├─ Frontend : React build served by Nginx, port 3001
             │     └─ Reverse proxy: Nginx on port 80 -> 3000
             │
             └──> EC2 Instance 2 (Ubuntu 24.04, t2.micro)
                   ├─ Backend : Node.js + Express, port 3000 (PM2)
                   └─ Reverse proxy: Nginx on port 80 -> 3000

                                    │
                                    ▼
                       MongoDB Atlas (M0, ap-south-1)
```

### Stack

| Layer            | Technology                             |
| ---------------- | -------------------------------------- |
| Frontend         | React (Create React App), built static |
| Backend          | Node.js, Express                       |
| Database         | MongoDB Atlas                          |
| Process manager  | PM2                                    |
| Web server       | Nginx (reverse proxy + static serving) |
| Compute          | AWS EC2 (t2.micro, Ubuntu 24.04)       |
| Load balancer    | AWS Application Load Balancer          |
| DNS              | Cloudflare (proxied)                   |
| Domain registrar | Namecheap (`saagarsaga.store`)         |

### Live endpoints

| Frontend (EC2 Instance 1)  | `http://13.201.222.34:3001`                                  |

## Setup process

### 1. Database — MongoDB Atlas

- Provisioned an Atlas M0 cluster (`Cluster0`) in the `ap-south-1` region to keep latency low to the EC2 instances.
- Created a dedicated database user `travelmemoryUser` with autogenerated credentials.
- Configured network access to allow inbound connections from the EC2 instances (`0.0.0.0/0` for development; intended to be tightened to specific EC2 IPs in production).
- Retrieved the SRV connection string and persisted it on each EC2 instance's `backend/.env` as `MONGO_URI` (kept out of version control).

### 2. Compute — AWS EC2 (two instances)

- Launched two Ubuntu 24.04 LTS `t2.micro` instances in `ap-south-1`: `travelmemory-backend-1` and `travelmemory-backend-2`.
- Created an SSH key pair (`travelmemory-key.pem`) and used it for both instances.
- Configured the security group with inbound rules for:
  - Port 22 (SSH)
  - Port 80 (HTTP)
  - Port 443 (HTTPS)
  - Port 3000 (Node backend, for direct testing)
  - Port 3001 (React static build served by Nginx on Instance 1)
- Both instances were provisioned identically so they could sit interchangeably behind the load balancer.

### 3. Application install (per EC2 instance)

- Installed `nodejs`, `npm`, `nginx`, and `git` via apt.
- Cloned this repository.
- Inside `backend/`:
  - `npm install`
  - Created `.env` with `MONGO_URI` and `PORT=3000`.
  - Installed PM2 globally and started the backend with `pm2 start index.js --name travelmemory-backend`.
- Inside `frontend/` (Instance 1 only):
  - Updated `src/url.js` to point at the deployed backend URL.
  - `npm install && npm run build` to produce the static `build/` directory.

### 4. Web server — Nginx

- On both instances, configured Nginx as a reverse proxy: requests to port 80 are forwarded to the Node backend on port 3000, with `Upgrade`, `Connection`, and `Host` headers preserved for WebSocket / proxy correctness.
- On Instance 1, added a second server block listening on port 3001 that serves the React `build/` directory with SPA fallback (`try_files $uri $uri/ /index.html`). File permissions on `/home/ubuntu` and the `build/` tree were adjusted (`chmod 755`) so that the `www-data` user could read them.
- Disabled the default Nginx site, ran `nginx -t` to validate, and restarted `nginx` via `systemctl`.

### 5. Load Balancer — AWS Application Load Balancer

- Created an internet-facing Application Load Balancer named `travelmemory-alb`, spanning all three availability zones in `ap-south-1` (`a`, `b`, `c`) for high availability.
- Reused the same security group as the EC2 instances so it could accept HTTP traffic on port 80.
- Created a target group `travelmemory-tg`:
  - Target type: instances
  - Protocol/port: HTTP / 80
  - Health check path: `/`
  - Registered both `travelmemory-backend-1` and `travelmemory-backend-2`.
- Configured the load balancer's HTTP:80 listener to forward to `travelmemory-tg`.

### 6. DNS — Cloudflare

- Added the domain `saagarsaga.store` to Cloudflare and noted the assigned nameservers.
- Configured DNS records:
  - `A` record on the root (`saagarsaga.store`) pointing at the Instance 1 public IP `13.201.222.34`.
  - `CNAME` record on `www` pointing at the load balancer's DNS name (`travelmemory-alb-1189099262.ap-south-1.elb.amazonaws.com`).
- Both records are proxied through Cloudflare for HTTPS termination, caching, and basic DDoS protection.

### 7. Domain registrar — Namecheap

- Switched the registrar's nameservers from Namecheap BasicDNS to Custom DNS, set to the Cloudflare-assigned pair (`aleena.ns.cloudflare.com`, `seamus.ns.cloudflare.com`) so DNS resolution flows through Cloudflare.

## Local Development

### Backend

Create a `.env` file in `backend/` with your MongoDB connection string:

```
MONGO_URI='ENTER_YOUR_URL'
PORT=3001
```

Then:

```bash
cd backend
npm install
npm start
```

### Frontend

Create a `.env` file in `frontend/` with the backend URL:

```bash
REACT_APP_BACKEND_URL=http://localhost:3001
```

Then:

```bash
cd frontend
npm install
npm start
```
